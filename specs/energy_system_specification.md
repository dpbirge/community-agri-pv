## System Configuration Structure

The system uses two configuration files:

`**energy_system_base.yaml**` defines the physical energy infrastructure:

- **Community solar** — ground-mounted PV arrays by density level, each with area in hectares
- **Agri-PV** — PV panels mounted over crop fields; defined per-field in `farm_profile_base.yaml`
via `condition: underpv_{density}`. Energy output is extracted by `src/energy_supply.py`
using the same per-hectare PV output data as community solar
- **Wind turbines** — by turbine type, each with a unit count
- **Diesel generator** — optional; `has_generator` flag, `type` matches the `type_id` column
in `generators-toy.csv` (e.g., `diesel_100kw`), `rated_capacity_kw` confirms the rating
- **Battery storage** — optional; `has_battery` flag, `type` matches the `type_id` column
in `batteries-toy.csv` (e.g., `lithium_iron_phosphate_medium`), `capacity_kwh` confirms size
- **Grid connection** — `grid_connection` field with values `full_grid` (bidirectional),
`limited_grid` (import-only or capped), or `off_grid`

`**energy_policy_base.yaml**` defines dispatch rules and constraints, separate from physical
infrastructure so strategies can be swapped independently. When policy parameters overlap
with equipment CSV defaults (e.g., battery SOC limits), the **policy value takes precedence**.
Equipment CSVs define hardware limits; the policy defines operational limits for the scenario.

### Grid Connection / Grid Mode Validation

The physical `grid_connection` field in `energy_system_base.yaml` constrains which
`grid.mode` values in `energy_policy_base.yaml` are valid. The energy balance module
must validate this at startup and raise an error for impossible combinations:

| `grid_connection` | Valid `grid.mode` values |
| --- | --- |
| `full_grid` | `full_grid`, `net_metering`, `feed_in_tariff`, `self_consumption` |
| `limited_grid` | `limited_grid`, `self_consumption`, `off_grid` |
| `off_grid` | `off_grid` |

When `grid.mode` is `limited_grid`, a non-null `monthly_import_cap_kwh` is required — the mode is defined by having a capped import.

Invalid combinations (e.g., `grid_connection: off_grid` with `grid.mode: full_grid`)
must raise `ValueError` with a descriptive message.

### Configuration Validation

Both `energy_system_base.yaml` and `energy_policy_base.yaml` are validated at runtime
by `_validate_energy_system_config()` and `_validate_energy_policy_config()` respectively,
called at the start of `compute_daily_energy_balance()`. Validation covers:

**Energy system config** — required top-level keys: `community_solar`, `wind_turbines`,
`generator`, `battery`, `grid_connection`. Generator section requires `has_generator`
(bool), and when true: `type` (str), `rated_capacity_kw` (positive number). Battery
section requires `has_battery` (bool), and when true: `type` (str), `capacity_kwh`
(positive number). `grid_connection` must be one of `full_grid`, `limited_grid`,
`off_grid`. Each `community_solar` entry must have `area_ha` (non-negative number).
Each `wind_turbines` entry must have `number` (non-negative integer).

**Energy policy config** — required top-level keys: `strategy`, `grid`, `cap_enforcement`,
`tariff`, `battery`, `generator`. `strategy` must be one of `minimize_cost`,
`minimize_grid_reliance`, `minimize_generator`. `grid.mode` must be one of `full_grid`,
`net_metering`, `feed_in_tariff`, `self_consumption`, `limited_grid`, `off_grid`.
Battery SOC fractions (`soc_min`, `soc_max`, `soc_initial`) must be in [0, 1] with
`soc_min < soc_max`. `generator.min_load_fraction` must be in (0, 1]. When `grid.mode` is `limited_grid`, `monthly_import_cap_kwh` must not be null.

All validation errors raise `ValueError` with a descriptive message identifying the
config file and the specific issue.

## Energy Balance Dispatch Model

The dispatch model takes daily renewable generation (from `src/energy_supply.py`) and daily
demand (from community demand + water balance) and allocates energy through available sources
according to the selected strategy. This is analogous to the central mixing tank in the water
system — the battery SOC [state of charge, percentage of capacity currently stored] is the
sole stateful element carried between days, parallel to tank volume/TDS.

### Dispatch Flow (per day)

1. **Compute net load**: `net_load = total_demand - total_renewable_generation`
2. **If net load <= 0** (renewable surplus):
  a. Charge battery with surplus (up to max charge rate and SOC ceiling)
   b. Export remaining surplus to grid (if grid mode permits and monthly export cap allows)
   c. Curtail any remaining surplus
3. **If net load > 0** (renewable deficit):
  a. Discharge battery to cover deficit (down to SOC floor)
   b. Import from grid to cover remaining deficit (up to monthly import cap daily allowance)
   c. Run diesel generator to cover remaining deficit (if available)
   d. Record any remaining unmet demand as energy deficit

### Battery State

- One state variable persists across days: SOC (kWh)
- **Daily timestep simplification**: the battery absorbs surplus and covers deficit within
each day without intra-day charge/discharge rate limits. At the daily timestep the battery
is assumed to handle moment-to-moment power flows during actual demand-supply fluctuations
without issue. The only constraints are SOC ceiling and floor.
- Charging: `energy_stored = min(surplus × charge_efficiency, soc_max_kwh - current_soc_kwh)`.
The accepted surplus consumed is `energy_stored / charge_efficiency`.
- Discharging: `energy_delivered = min(deficit, available_stored × discharge_efficiency)`
where `available_stored = current_soc_kwh - soc_min_kwh`. The SOC draw is
`energy_delivered / discharge_efficiency`.
- When `has_battery: false`, battery steps are skipped (surplus goes directly to grid/curtailment,
deficit goes directly to grid/generator)
- Cycle tracking is intentionally omitted — the daily timestep does not provide sufficient
resolution for meaningful cycle counting (partial cycles, depth-of-discharge weighting)
- **Policy overrides CSV**: SOC limits (`soc_min`, `soc_max`, `soc_initial`) from
`energy_policy_base.yaml` take precedence over the equipment CSV's `soc_min`/`soc_max`
columns. The CSV values represent hardware limits; the policy values are the operational
limits chosen for this scenario.

### Generator Dispatch

- Generator runs only when battery and grid cannot meet the remaining deficit
- Minimum load constraint: generators cannot operate below `min_load_fraction × rated_capacity`
without risking engine damage. If the remaining deficit requires less than minimum load power,
the generator runs at minimum load for a minimum runtime of 6 hours (representing a single
operational shift — startup, run, shutdown). The generator produces
`min_load_kw × runtime_hours` kWh; any output beyond the deficit is excess energy
that charges the battery (cycle charging) or is curtailed. When the deficit exceeds minimum
load output, `power_kw = max(deficit_kwh / 24, min_load_kw)` capped at `rated_capacity_kw`,
and `hours = deficit_kwh / power_kw`, capped at 24 hours.
- Fuel consumption uses the Willans line model from `generators-toy.csv`:
`fuel_liters = (a × rated_kw + b × power_kw) × hours_of_operation`
where `a` = `sfc_coefficient_a` (no-load coefficient) and `b` = `sfc_coefficient_b`
(incremental coefficient).
- When `has_generator: false`, generator steps are skipped and unmet demand becomes deficit
- **Generator min-load excess and battery renewable tracking**: when the generator runs at
minimum load, excess energy beyond the deficit is offered to the battery first, then
curtailed. This generator-sourced charge is the only non-renewable energy that enters the
battery. All other battery charge comes from renewable surplus. The
`battery_renewable_fraction` state variable tracks what share of stored energy is
renewable-origin: it is updated on each charge (weighted blend of current fraction and
source fraction) and read on discharge to compute `renewable_fraction` metrics.

## Policy & Dispatch

Policy is configured in `energy_policy_base.yaml`. The dispatch loop above is parameterized
by a strategy string that controls source priority. Three strategies cover the practical
range for community agri-PV systems:

### `minimize_cost` (default)

Merit-order dispatch — sources used in ascending order of marginal cost. This is the standard
economic dispatch approach used in utility-scale and microgrid controllers (HOMER Pro default).

- **Surplus priority**: battery charge → grid export → curtail
- **Deficit priority**: battery discharge → grid import → diesel generator
- **Objective**: lowest daily energy cost

### `minimize_grid_reliance`

Prioritizes using locally generated energy and minimizing grid dependence. Standard approach
for systems with unreliable grid connections or self-consumption regulatory frameworks.

- **Surplus priority**: battery charge → grid export → curtail (same as other strategies; surplus energy is not wasted when grid export is permitted by the grid mode)
- **Deficit priority**: battery discharge → diesel generator → grid import (grid is last resort)
- **Objective**: highest self-sufficiency ratio (differentiated by deficit order, not surplus order)

### `minimize_generator`

Minimizes diesel fuel use and generator runtime. Prefers grid import over generator dispatch
to reduce fuel cost and emissions, accepting higher grid dependence.

- **Surplus priority**: battery charge → grid export → curtail
- **Deficit priority**: battery discharge → grid import (exhaust monthly cap daily allowance before generator) → diesel generator
- **Objective**: fewest generator runtime hours and lowest fuel consumption

### Grid Interaction Modes

The `grid` section in the energy policy controls how the system interacts with the external
grid. Mode determines import/export availability:


| Mode               | Import | Export   | Description                                   |
| ------------------ | ------ | -------- | --------------------------------------------- |
| `full_grid`        | yes    | yes      | Unrestricted bidirectional exchange           |
| `net_metering`     | yes    | credited | Export credited against import at retail rate |
| `feed_in_tariff`   | yes    | sold     | Export purchased at a fixed FIT rate          |
| `self_consumption` | yes    | no       | Import allowed, export blocked                |
| `limited_grid`     | capped | no       | Import up to monthly cap, no export           |
| `off_grid`         | no     | no       | No grid connection                            |


Grid mode interacts with strategy: in `off_grid` mode, all grid steps are skipped regardless
of strategy. In `self_consumption` mode, surplus energy that exceeds battery capacity is
curtailed rather than exported.

### Monthly Grid Cap Enforcement

All grid caps operate at the **monthly** timescale, consistent with the water module's
monthly cap pattern. Both `monthly_import_cap_kwh` and `monthly_export_cap_kwh` in
`energy_policy_base.yaml` are monthly totals.

When `monthly_import_cap_kwh` or `monthly_export_cap_kwh` is set:

- `look_ahead: true` — remaining monthly cap divided evenly across remaining days in the
month. Daily allowance = `(monthly_cap - used_this_month) / days_remaining_in_month`.
- `look_ahead: false` — draw or export freely until the monthly cap is exhausted.

Monthly state resets at the start of each calendar month, following the same
`(year, month)` boundary pattern used in `src/water.py`.

### Tariff Assignment

Grid electricity costs are split by demand category to reflect Egypt's tiered tariff structure:

- **Water system energy** (pumping, treatment, irrigation application) — charged at the
**agricultural** rate from `historical_grid_electricity_prices-toy.csv`. Uses the
`usd_per_kwh_avg_daily` column (daily average, consistent with the daily timestep).
- **Community building energy** (households, offices, warehouses, cold storage) — charged at
the **commercial** rate from `historical_grid_electricity_prices_unsubsidized-toy.csv`.

The `tariff` section in `energy_policy_base.yaml` maps each demand category to a registry key
under `electricity_pricing`. The energy balance module looks up the rate using the simulation
date. **Price data is monthly** — the module loads each tariff CSV, parses the `date` column,
and uses `pd.merge_asof` (or equivalent forward-fill) to map each simulation day to the
most recent monthly rate. This produces one scalar rate per day per tariff category, which
is passed to `_dispatch_day()`.

### Net Metering Accounting

When `grid.mode: net_metering`:

1. **Monthly energy netting** — grid export kWh are offset against grid import kWh within
each calendar month. The module tracks cumulative monthly import and export.
2. **Monthly settlement** — at each calendar month boundary, accumulators reset. No credits
carry forward between months. Monthly export that exceeds monthly import is unpaid energy
sent to the grid — economically equivalent to curtailment from the community's perspective.
3. **Implementation at daily timestep** — daily `grid_import_cost` is computed incrementally
from the change in the monthly net billable position. Each day, the billable position is
`max(0, monthly_import - monthly_export)`. The daily cost is the incremental increase in
the billable position × weighted tariff rate. On days when exports reduce the net
position, cost is zero. This ensures the sum of daily costs equals the correct monthly
net-metered bill.

### Feed-In Tariff Accounting

When `grid.mode: feed_in_tariff`, all exported energy is purchased at the rate specified in
`grid.export_rate_usd_kwh`. If null, the rate is looked up from `feed_in_tariff_rates-toy.csv`
using `grid.capacity_tier` to match the `project_size_category` column. The
`rate_usd_kwh_effective` column provides the unified USD rate for all tiers (EGP-denominated
rows are pre-converted at their inception-date exchange rate in the CSV).

## Energy Policy Configuration

```yaml
# settings/energy_policy_base.yaml

energy_policy_name: baseline_energy_policy

# --- Dispatch Strategy ---
# minimize_cost:                merit-order by marginal cost (renewables > battery > grid > generator)
# minimize_grid_reliance:    minimize grid reliance (renewables > battery > generator > grid)
# minimize_generator:           minimize diesel use (renewables > battery > grid > generator)
strategy: minimize_cost

# --- Grid Interaction ---
grid:
  mode: self_consumption    # full_grid | net_metering | feed_in_tariff | self_consumption
                            # | limited_grid | off_grid
  monthly_import_cap_kwh: null  # null = unlimited; max monthly grid import (kWh)
  monthly_export_cap_kwh: null  # null = unlimited; max monthly grid export (kWh)
  export_rate_usd_kwh: null     # explicit FIT rate override; null = look up from CSV by capacity_tier
  capacity_tier: null           # project_size_category in feed_in_tariff_rates CSV
                                # (e.g., household, commercial_small, commercial_large, utility_scale)
                                # only used when mode = feed_in_tariff and export_rate_usd_kwh is null

# --- Grid Cap Enforcement ---
cap_enforcement:
  look_ahead: true    # true = spread remaining monthly cap evenly over remaining days
                      # false = draw freely until monthly cap exhausted

# --- Tariff Assignment ---
# Agricultural rate applies to water system electricity (pumping, treatment,
# irrigation application). Commercial rate applies to community building
# electricity (households, offices, warehouses, cold storage).
tariff:
  water_energy: agricultural     # registry key under electricity_pricing
  building_energy: commercial    # registry key under electricity_pricing

# --- Net Metering ---
# Monthly energy netting: export kWh offset against import kWh within each
# calendar month. No credit carries forward between months. Monthly export
# exceeding import is unpaid (equivalent to curtailment economically).
net_metering:
  credit_period: monthly         # credits accumulate and settle monthly

# --- Battery Dispatch ---
# Only active when energy_system has_battery: true.
# SOC limits here override the equipment CSV defaults for this scenario.
battery:
  soc_min: 0.20             # minimum state of charge (fraction of capacity)
  soc_max: 0.95             # maximum state of charge
  soc_initial: 0.50         # starting SOC for simulation

# --- Generator Dispatch ---
# Only active when energy_system has_generator: true.
# Generator runs only when battery and grid cannot meet remaining deficit.
generator:
  min_load_fraction: 0.30   # minimum operating load as fraction of rated capacity
```

### Metrics


| Column                   | Description                                                              |
| ------------------------ | ------------------------------------------------------------------------ |
| `self_sufficiency_ratio` | `(total_demand - grid_import - deficit) / total_demand`                  |
| `self_consumption_ratio` | `(renewable_consumed + renewable_battery_charge) / total_renewable`      |
| `renewable_fraction`     | `(renewable_consumed + battery_discharge_renewable_kwh) / total_demand`  |

`battery_discharge_renewable_kwh` is computed as
`battery_discharge_kwh × battery_renewable_fraction` (the renewable-origin share of
discharged energy). At the daily timestep, all battery charge from surplus is
renewable-origin. The only non-renewable charge source is generator min-load excess
(cycle charging). When the battery is absent, `renewable_fraction` equals
`renewable_consumed / total_demand`.

> **Simplification**: renewable energy stored in the battery is counted as self-consumed
> at charge time, rather than tracked through discharge across multiple days. This avoids
> the complexity of tracking energy tranches day-to-day while producing materially correct
> results over multi-year simulation periods. The only non-renewable battery charge source
> is generator min-load excess (cycle charging).

Self-sufficiency ratio measures the fraction of demand met by all local microgrid
resources (renewables + battery + generator) without grid import. Generator output
counts as self-sufficient because it is locally dispatched, even though it depends on
external fuel supply. Unmet demand (deficit) is excluded.
Renewable fraction measures the fraction of demand
ultimately sourced from wind and solar, including the renewable share of battery discharge.

## Energy Balance Module

`src/energy_balance.py` composes energy generation, community demand, and water system energy
into a single daily DataFrame. This is analogous to `src/water_balance.py`.

### Module API

```python
compute_daily_energy_balance(
    energy_config_path,          # settings/energy_system_base.yaml
    energy_policy_path,          # settings/energy_policy_base.yaml
    community_config_path,       # settings/community_demands_base.yaml
    farm_profiles_path,          # settings/farm_profile_base.yaml
    registry_path,               # settings/data_registry_base.yaml
    *,
    water_balance_df=None,       # from compute_daily_water_balance(); provides water energy
    root_dir=None,
)
```

Returns a DataFrame with all columns listed above.

### Date Range Alignment

The energy balance module must align date ranges across its input sources:

- **Energy generation** (2010–2025): produced by `compute_daily_energy()`
- **Community demand** (2010–2025): produced by `compute_daily_demands()`
- **Water balance** (scenario period): pre-computed, passed as `water_balance_df`
- **Price data** (2010–2024): monthly CSVs forward-filled to daily

When `water_balance_df` is provided, the simulation period is the intersection of the
water balance date range and the energy generation date range. When `water_balance_df`
is `None` (energy-only analysis without water), the simulation period is the full
energy generation date range, and `water_energy_demand_kwh` is zero for all days.

Price data is forward-filled: if the simulation extends beyond the last price entry,
the final available rate is used. If the simulation starts before the first price entry,
the module raises `ValueError`.

### Orchestration

1. Call `compute_daily_energy()` → renewable generation DataFrame
2. Call `compute_daily_demands()` → community energy demand DataFrame
3. Accept `water_balance_df` (pre-computed) → extract `total_water_energy_kwh` column
4. Compute date intersection; filter all DataFrames to the common date range
5. Load configs; validate energy system and policy configs
   (`_validate_energy_system_config`, `_validate_energy_policy_config`)
6. Load price data (agricultural tariff, commercial tariff, diesel) from registry;
   forward-fill monthly → daily using `pd.merge_asof` or `reindex(method='ffill')`
7. Load FIT rate if `grid.mode == feed_in_tariff`: use `grid.export_rate_usd_kwh` if
   set, else look up `rate_usd_kwh_effective` from FIT CSV by `grid.capacity_tier`
8. Pre-compute `battery_specs` and `generator_specs` dicts from equipment CSVs + policy
9. Validate `grid_connection` vs `grid.mode` compatibility (see validation table above)
10. Run `_run_simulation()` which calls `_dispatch_day()` for each day in date order,
    carrying battery SOC and monthly cap/metering state forward
11. Assemble and return the unified DataFrame via `_order_energy_balance_columns()`

### Internal Dispatch Function

```python
_dispatch_day(
    day,                        # date for this row
    total_demand_kwh,           # community + water energy demand for the day
    community_demand_kwh,       # household + building demand (for tariff split)
    water_demand_kwh,           # pumping + treatment + application demand (for tariff split)
    total_renewable_kwh,        # total solar + wind generation
    total_solar_kwh,            # solar subtotal (pass-through to output)
    total_wind_kwh,             # wind subtotal (pass-through to output)
    battery_specs,              # dict or None; pre-computed from CSV + policy
    generator_specs,            # dict or None; pre-computed from CSV + policy
    battery_state,              # mutable carry-forward: {'soc_kwh': float, 'renewable_fraction': float}
    strategy,                   # dispatch strategy string (e.g. 'minimize_cost')
    grid_mode,                  # grid interaction mode string (e.g. 'self_consumption')
    grid_cap_state,             # monthly grid import/export cap tracking
    net_metering_state,         # monthly net metering tracking
    ag_tariff,                  # agricultural electricity rate for this day (USD/kWh)
    commercial_tariff,          # commercial electricity rate for this day (USD/kWh)
    diesel_price,               # diesel fuel price for this day (USD/liter)
    export_rate,                # grid export compensation rate (0.0 when no export)
)
```

Policy parameters are decomposed into scalar arguments by `_run_simulation()` before
being passed to `_dispatch_day()`. This keeps the dispatch function stateless with no
I/O and no coupling to the policy YAML structure.

Returns `(row_dict, battery_state)`. The row dict contains all output columns for this
day; `battery_state` carries SOC and renewable fraction forward to the next day. This
mirrors the `(row, tank)` return pattern from `_dispatch_day()` in `src/water.py`.

**Key design decisions:**

- **`battery_specs`/`generator_specs` are `None` when absent** — dispatch code checks
`is not None` to skip equipment steps, rather than checking `has_battery`/`has_generator`
flags. Pre-computed by the caller from equipment CSVs + policy SOC limits:
`battery_specs` contains `capacity_kwh`, `soc_min_kwh`, `soc_max_kwh`,
`charge_efficiency`, `discharge_efficiency`. No charge/discharge rate limits —
at the daily timestep the SOC ceiling and floor are the only constraints. `generator_specs` contains `rated_capacity_kw`, `min_load_kw`
(= `min_load_fraction × rated_capacity_kw`), `sfc_coefficient_a`, `sfc_coefficient_b`.
- **Equipment CSV lookup uses `type_id`** — the config `type` field (e.g., `diesel_100kw`)
matches the `type_id` column in the equipment CSV. Single-column lookup, no composite keys.
- **Policy SOC limits override CSV SOC limits** — `battery_specs.soc_min_kwh` and
`soc_max_kwh` are computed from the policy's `battery.soc_min` and `battery.soc_max`
fractions × `capacity_kwh`, not from the CSV's `soc_min`/`soc_max` columns.
- **Prices are scalar arguments** — the caller performs date-indexed lookups from price
DataFrames and passes the day's values. Keeps `_dispatch_day()` stateless with no I/O.
- **Monthly tracking lives in the caller** (`_run_simulation`), not in `_dispatch_day()`.
Grid cap and net metering accumulators are updated by the caller after each
day's row is returned. The dispatch function receives read-only snapshots.
- **Strategy branching uses ordered action lists** — surplus and deficit priority orders
are defined as lists of action strings (`['battery', 'export', 'curtail']`), then
iterated. This keeps dispatch logic DRY and makes adding strategies trivial.

renewable_consumed = min(total_renewable, total_demand)

if net_load <= 0:  # SURPLUS
    surplus = -net_load
    for action in surplus_priority_order[strategy]:
        if action == 'battery':   _charge_battery(..., renewable=True)
        if action == 'export':    grid_export (if mode allows, up to monthly cap daily allowance)
        if action == 'curtail':   curtailed_kwh = remaining

if net_load > 0:   # DEFICIT
    deficit = net_load
    for action in deficit_priority_order[strategy]:
        if action == 'battery':   _discharge_battery(...)
        if action == 'grid':      grid_import (up to monthly cap daily allowance)
        if action == 'generator': _run_generator(...)
            → min-load excess: _charge_battery(..., renewable=False), then curtailed
    deficit_kwh = remaining unmet demand

cost: weighted tariff import + fuel cost − export revenue
metrics: SSR, SCR, renewable_fraction

### Assumptions

- Daily timestep: generation and demand are daily totals, not hourly profiles. Battery
charge/discharge is computed from daily surplus/deficit — no intra-day scheduling
- Renewable generation is non-curtailable by default; curtailment occurs only when surplus
exceeds battery + export capacity
- Grid tariff uses daily average rate (`usd_per_kwh_avg_daily` column); agricultural rate
for water energy, commercial rate for building energy. Price data is monthly and
forward-filled to daily resolution.
- Generator startup cost and minimum runtime are not modeled at the daily timestep
- Battery cycle tracking and degradation are not modeled — the daily timestep does not
provide sufficient resolution for meaningful cycle counting

## Intentionally Deferred Energy Extensions

The following capabilities were evaluated and excluded from the current scope. They may be
revisited if the simulation objectives expand.

- **Hourly dispatch / intra-day scheduling** — Splitting the daily timestep into hourly or
sub-hourly intervals to model solar peak vs evening demand curves. Requires hourly
generation and demand profiles and significantly increases simulation complexity. The daily
timestep is sufficient for seasonal energy balance and policy comparison.
- **Inverter efficiency curves** — Modeling inverter losses as a function of load factor rather
than a flat system loss. Adds accuracy at partial load but requires detailed inverter specs
and hourly resolution to be meaningful.
- **Battery degradation and cycle tracking** — Reducing usable battery capacity over time based
on cycle count, depth of discharge, and temperature. Requires sub-daily resolution for
meaningful cycle counting; the daily timestep cannot distinguish partial from full cycles.
Relevant for multi-year financial analysis but adds complexity with minimal impact on annual
energy balance.
- **Demand response / load shedding** — Deferring or curtailing non-critical loads (e.g.,
warehouse climate control) during deficit periods. Requires load priority metadata and a
controllable-load sub-model.
- **Time-of-use tariff optimization** — Shifting battery charge/discharge to exploit
off-peak/peak price differentials. Requires hourly tariff schedules and intra-day dispatch,
which conflicts with the daily timestep.
- **Electric vehicle charging** — Farm EVs or community transport as both load and mobile
storage (V2G). Adds a vehicle fleet model and charging schedule layer.
- **Hydrogen electrolysis / fuel cell** — Using surplus renewable energy to produce hydrogen
for long-term storage or fuel cell generation. Adds an entirely new energy carrier and
storage pathway.
- **Peer-to-peer energy trading** — Inter-household or inter-farm energy trading within the
community microgrid. Requires institutional and market mechanism modeling beyond physical
simulation.
- **Predictive dispatch / weather forecasting** — Using day-ahead weather forecasts to
pre-position battery SOC. Adds a forecast model layer and uncertainty handling.
