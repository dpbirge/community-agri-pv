## System Configuration Structure

The system uses two configuration files:

`**energy_system_base.yaml**` defines the physical energy infrastructure:

- **Community solar** — ground-mounted PV arrays by density level, each with area in hectares
- **Agri-PV** — PV panels mounted over crop fields; defined per-field in `farm_profile_base.yaml`  
via `condition: underpv_{density}`. Energy output is extracted by `src/energy_supply.py`  
using the same per-hectare PV output data as community solar
- **Wind turbines** — by turbine type, each with a unit count
- **Diesel generator** — optional; `has_generator` flag, `type` selects a row from
`generators-toy.csv` (e.g., `diesel_100kw`), `rated_capacity_kw` confirms the rating
- **Battery storage** — optional; `has_battery` flag, `type` selects a row from
`batteries-toy.csv` (e.g., `lithium_iron_phosphate_medium`), `capacity_kwh` confirms size
- **Grid connection** — `grid_connection` field with values `full_grid` (bidirectional),
`limited_grid` (import-only or capped), or `off_grid`

`**energy_policy_base.yaml**` defines dispatch rules and constraints, separate from physical
infrastructure so strategies can be swapped independently.

## Energy Balance Dispatch Model

The dispatch model takes daily renewable generation (from `src/energy_supply.py`) and daily  
demand (from community demand + water balance) and allocates energy through available sources  
according to the selected strategy. This is analogous to the central mixing tank in the water  
system — the battery SOC [state of charge, percentage of capacity currently stored] is the  
sole stateful element carried between days, parallel to tank volume/TDS.

### Dispatch Flow (per day)

1. **Compute net load**: `net_load = total_demand - total_renewable_generation`
2. **If net load <= 0** (renewable surplus):
  a. Charge battery with surplus (up to max charge rate and SOC ceiling)  
   b. Export remaining surplus to grid (if grid mode permits)  
   c. Curtail any remaining surplus
3. **If net load > 0** (renewable deficit):
  a. Discharge battery to cover deficit (down to SOC floor)  
   b. Import from grid to cover remaining deficit (up to import cap)  
   c. Run diesel generator to cover remaining deficit (if available)  
   d. Record any remaining unmet demand as energy deficit

### Battery State

- One state variable persists across days: SOC (kWh)
- Charging: `energy_stored = surplus × charge_efficiency`, capped by `max_charge_rate_kw × 24`
and `(soc_max - current_soc) × capacity_kwh`
- Discharging: `energy_delivered = stored_energy × discharge_efficiency`, capped by
`max_discharge_rate_kw × 24` and `(current_soc - soc_min) × capacity_kwh`
- When `has_battery: false`, battery steps are skipped (surplus goes directly to grid/curtailment,
deficit goes directly to grid/generator)
- Cycle tracking is intentionally omitted — the daily timestep does not provide sufficient
resolution for meaningful cycle counting (partial cycles, depth-of-discharge weighting)

### Generator Dispatch

- Generator runs only when battery and grid cannot meet the remaining deficit
- Minimum load constraint: if required output < `min_load_fraction × rated_capacity`,  
generator runs at minimum load; excess charges the battery (cycle charging) or is curtailed
- Fuel consumption uses the Willans line model from `generators-toy.csv`:  
`fuel_liters = a × power_kw + b × rated_kw` per hour of operation
- When `has_generator: false`, generator steps are skipped and unmet demand becomes deficit

## Policy & Dispatch

Policy is configured in `energy_policy_base.yaml`. The dispatch loop above is parameterized  
by a strategy string that controls source priority. Three strategies cover the practical  
range for community agri-PV systems:

### `minimize_cost` (default)

Merit-order dispatch — sources used in ascending order of marginal cost. This is the standard  
economic dispatch approach used in utility-scale and microgrid controllers (HOMER Pro default).

- **Surplus priority**: battery charge → grid export → curtail
- **Deficit priority**: battery discharge → grid import → diesel generator
- **Objective**: lowest daily energy cost

### `maximize_self_consumption`

Prioritizes using locally generated energy and minimizing grid dependence. Standard approach  
for systems with unreliable grid connections or self-consumption regulatory frameworks.

- **Surplus priority**: battery charge → curtail (export avoided or minimized)
- **Deficit priority**: battery discharge → diesel generator → grid import (grid is last resort)
- **Objective**: highest self-sufficiency ratio

### `minimize_generator`

Minimizes diesel fuel use and generator runtime. Prefers grid import over generator dispatch  
to reduce fuel cost and emissions, accepting higher grid dependence.

- **Surplus priority**: battery charge → grid export → curtail
- **Deficit priority**: battery discharge → grid import (exhaust grid cap before generator) → diesel generator
- **Objective**: fewest generator runtime hours and lowest fuel consumption

### Grid Interaction Modes

The `grid` section in the energy policy controls how the system interacts with the external  
grid. Mode determines import/export availability:


| Mode               | Import | Export   | Description                                   |
| ------------------ | ------ | -------- | --------------------------------------------- |
| `full_grid`        | yes    | yes      | Unrestricted bidirectional exchange           |
| `net_metering`     | yes    | credited | Export credited against import at retail rate |
| `feed_in_tariff`   | yes    | sold     | Export purchased at a fixed FIT rate          |
| `self_consumption` | yes    | no       | Import allowed, export blocked                |
| `limited_grid`     | capped | no       | Import up to daily cap, no export             |
| `off_grid`         | no     | no       | No grid connection                            |


Grid mode interacts with strategy: in `off_grid` mode, all grid steps are skipped regardless  
of strategy. In `self_consumption` mode, surplus energy that exceeds battery capacity is  
curtailed rather than exported.

### Monthly Grid Cap Enforcement

When `grid_import_cap_kwh` is set (analogous to municipal water monthly cap):

- `look_ahead: true` — remaining monthly cap divided evenly across remaining days
- `look_ahead: false` — draw freely until cap is exhausted

### Tariff Assignment

Grid electricity costs are split by demand category to reflect Egypt's tiered tariff structure:

- **Water system energy** (pumping, treatment, irrigation application) — charged at the
**agricultural** rate from `historical_grid_electricity_prices-toy.csv`. Uses the
`usd_per_kwh_avg_daily` column (daily average, consistent with the daily timestep).
- **Community building energy** (households, offices, warehouses, cold storage) — charged at
the **commercial** rate from `historical_grid_electricity_prices_unsubsidized-toy.csv`.

The `tariff` section in `energy_policy_base.yaml` maps each demand category to a registry key
under `electricity_pricing`. The energy balance module looks up the daily rate from the
appropriate price file using the simulation date.

### Net Metering Accounting

When `grid.mode: net_metering`, the following Egypt EgyptERA rules apply:

1. **Monthly credit accumulation** — grid export kWh are credited against grid import kWh
within each calendar month. Net import for the month determines the bill.
2. **Annual settlement** — at the end of June (fiscal year), any remaining credit balance
is purchased by the distribution company at the EETC solar PPA rate
(`excess_buyback_usd_kwh`, default 0.04 USD/kWh). This rate is well below retail tariff,
incentivizing self-consumption over overproduction.
3. **Implementation at daily timestep** — the module tracks cumulative monthly import and
export. Daily `grid_import_cost` uses the net monthly position: if cumulative monthly
export exceeds cumulative monthly import, daily import cost is zero (covered by credits).
At month boundaries, unused credits roll forward. At June settlement, excess credits are
converted to revenue at the buyback rate.

### Feed-In Tariff Accounting

When `grid.mode: feed_in_tariff`, all exported energy is purchased at the rate specified in
`grid.export_rate_usd_kwh`. If null, the rate is looked up from `feed_in_tariff_rates-toy.csv`
based on the system's capacity tier.

## Energy Policy Configuration

```yaml
# settings/energy_policy_base.yaml

energy_policy_name: baseline_energy_policy

# --- Dispatch Strategy ---
# minimize_cost:                merit-order by marginal cost (renewables > battery > grid > generator)
# maximize_self_consumption:    minimize grid import (renewables > battery > generator > grid)
# minimize_generator:           minimize diesel use (renewables > battery > grid > generator)
strategy: minimize_cost

# --- Grid Interaction ---
grid:
  mode: self_consumption    # full_grid | net_metering | feed_in_tariff | self_consumption
                            # | limited_grid | off_grid
  import_cap_kwh: null      # null = unlimited; max daily grid import (kWh)
  export_rate_usd_kwh: null # feed-in tariff rate; null when mode != feed_in_tariff

# --- Grid Cap Enforcement ---
cap_enforcement:
  look_ahead: true

# --- Tariff Assignment ---
# Agricultural rate applies to water system electricity (pumping, treatment,
# irrigation application). Commercial rate applies to community building
# electricity (households, offices, warehouses, cold storage).
tariff:
  water_energy: agricultural     # registry key under electricity_pricing
  building_energy: commercial    # registry key under electricity_pricing

# --- Net Metering ---
# Egypt EgyptERA rules: monthly credit offset against consumption, annual
# settlement in June. Excess beyond annual offset purchased at EETC solar
# PPA price (~0.04 USD/kWh).
net_metering:
  credit_period: monthly         # credits accumulate monthly
  settlement_period: annual      # annual true-up in June
  settlement_month: 6            # June fiscal year end
  excess_buyback_usd_kwh: 0.04  # EETC solar PPA approximate rate

# --- Battery Dispatch ---
# Only active when energy_system has_battery: true
battery:
  soc_min: 0.20             # minimum state of charge (fraction of capacity)
  soc_max: 0.95             # maximum state of charge
  soc_initial: 0.50         # starting SOC for simulation

# --- Generator Dispatch ---
# Only active when energy_system has_generator: true
generator:
  soc_start_threshold: 0.25 # start generator when battery SOC drops below this
  min_load_fraction: 0.30   # minimum operating load as fraction of rated capacity
  monthly_fuel_cap_liters: null  # null = unlimited
```

## Energy Balance Module

`src/energy_balance.py` composes energy generation, community demand, and water system energy  
into a single daily DataFrame. This is analogous to `src/water_balance.py`.

### Module API

```python
compute_daily_energy_balance(
    energy_config_path,          # settings/energy_system_base.yaml
    energy_policy_path,          # settings/energy_policy_base.yaml
    community_config_path,       # settings/community_demands_base.yaml
    farm_profiles_path,          # settings/farm_profile_base.yaml
    registry_path,               # settings/data_registry_base.yaml
    *,
    water_balance_df=None,       # from compute_daily_water_balance(); provides water energy
    root_dir=None,
)
```

Returns a DataFrame with all columns listed above.

### Orchestration

1. Call `compute_daily_energy()` → renewable generation DataFrame
2. Call `compute_daily_demands()` → community energy demand DataFrame
3. Accept `water_balance_df` (pre-computed) → extract `total_water_energy_kwh` column
4. Load price data (agricultural tariff, commercial tariff, diesel) from registry
5. Pre-compute `battery_specs` and `generator_specs` dicts from equipment CSVs + policy
6. Run `_dispatch_day()` for each day in date order, carrying battery SOC forward
7. Accumulate monthly grid cap, fuel cap, and net metering state in the caller loop
8. Assemble and return the unified DataFrame

### Internal Dispatch Function

```python
_dispatch_day(
    total_demand_kwh,           # community + water energy demand for the day
    community_demand_kwh,       # household + building demand (for tariff split)
    water_demand_kwh,           # pumping + treatment + application demand (for tariff split)
    total_renewable_kwh,        # total solar + wind generation
    total_solar_kwh,            # solar subtotal (pass-through to output)
    total_wind_kwh,             # wind subtotal (pass-through to output)
    battery_specs,              # dict or None; pre-computed from CSV + policy
    generator_specs,            # dict or None; pre-computed from CSV + policy
    battery_state,              # mutable carry-forward: {'soc_kwh': float}
    policy,                     # parsed energy_policy dict (strategy, grid_mode, etc.)
    grid_cap_state,             # monthly grid import cap tracking
    fuel_cap_state,             # monthly fuel cap tracking
    net_metering_state,         # monthly net metering credit tracking
    ag_tariff_usd_kwh,          # agricultural electricity rate for this day
    commercial_tariff_usd_kwh,  # commercial electricity rate for this day
    diesel_price_usd_liter,     # diesel fuel price for this day
    export_rate_usd_kwh,        # grid export compensation rate (0.0 when no export)
)
```

Returns `(row_dict, battery_state)`. The row dict contains all output columns for this
day; `battery_state` carries SOC forward to the next day. This mirrors the `(row, tank)`
return pattern from `_dispatch_day()` in `src/water.py`.

**Key design decisions:**

- **`battery_specs`/`generator_specs` are `None` when absent** — dispatch code checks
`is not None` to skip equipment steps, rather than checking `has_battery`/`has_generator`
flags. Pre-computed by the caller from equipment CSVs + policy SOC limits:
`battery_specs` contains `capacity_kwh`, `soc_min_kwh`, `soc_max_kwh`,
`max_charge_kwh` (= `power_kw × 24`), `max_discharge_kwh`, `charge_efficiency`,
`discharge_efficiency`. `generator_specs` contains `rated_capacity_kw`, `min_load_kw`
(= `min_load_fraction × rated_capacity_kw`), `sfc_coefficient_a`, `sfc_coefficient_b`.
- **Prices are scalar arguments** — the caller performs date-indexed lookups from price
DataFrames and passes the day's values. Keeps `_dispatch_day()` stateless with no I/O.
- **Monthly tracking lives in the caller** (`_run_simulation`), not in `_dispatch_day()`.
Grid cap, fuel cap, and net metering accumulators are updated by the caller after each
day's row is returned. The dispatch function receives read-only snapshots.
- **Strategy branching uses ordered action lists** — surplus and deficit priority orders
are defined as lists of action strings (`['battery', 'export', 'curtail']`), then
iterated. This keeps dispatch logic DRY and makes adding strategies trivial.

**Helper functions:**

- `_charge_battery(surplus_kwh, battery_specs, battery_state)` → `(accepted_kwh, stored_kwh)`
- `_discharge_battery(deficit_kwh, battery_specs, battery_state)` → `(delivered_kwh, soc_draw_kwh)`
- `_grid_import_available(grid_mode, grid_cap_state)` → `float` (daily allowance)
- `_grid_export_allowed(grid_mode)` → `bool`
- `_run_generator(deficit_kwh, generator_specs, fuel_cap_state)` →
`(delivered_kwh, excess_kwh, fuel_liters, runtime_hours)`
- `_compute_grid_import_cost(import_kwh, community_kwh, water_kwh, total_kwh,
ag_tariff, commercial_tariff)` → `float` (weighted by demand share)
- `_compute_net_metering_cost(...)` → `float` (credit-adjusted daily cost)
- `_init_energy_dispatch_row(...)` → zeroed row dict with all output keys
- `_finalize_energy_dispatch_row(row, battery_specs, battery_state)` → computes
metrics (SSR, SCR, renewable fraction) and cost totals

**Dispatch pseudocode:**

```python
net_load = total_demand - total_renewable
renewable_consumed = min(total_renewable, total_demand)

if net_load <= 0:  # SURPLUS
    surplus = -net_load
    for action in surplus_priority_order[strategy]:
        if action == 'battery':   _charge_battery(...)
        if action == 'export':    grid_export (if mode allows)
        if action == 'curtail':   curtailed_kwh = remaining

if net_load > 0:   # DEFICIT
    deficit = net_load
    for action in deficit_priority_order[strategy]:
        if action == 'battery':   _discharge_battery(...)
        if action == 'grid':      grid_import (up to daily cap)
        if action == 'generator': _run_generator(...)
            → min-load excess charges battery, then curtailed
    deficit_kwh = remaining unmet demand

cost: weighted tariff import + fuel cost − export revenue
metrics: SSR, SCR, renewable_fraction
```

### Assumptions

- Daily timestep: generation and demand are daily totals, not hourly profiles. Battery  
charge/discharge is computed from daily surplus/deficit — no intra-day scheduling
- Renewable generation is non-curtailable by default; curtailment occurs only when surplus  
exceeds battery + export capacity
- Grid tariff uses daily average rate; agricultural rate for water energy, commercial rate
for building energy (configured in `energy_policy_base.yaml` tariff section)
- Generator startup cost and minimum runtime are not modeled at the daily timestep
- Battery cycle tracking and degradation are not modeled — the daily timestep does not
provide sufficient resolution for meaningful cycle counting

## Intentionally Deferred Energy Extensions

The following capabilities were evaluated and excluded from the current scope. They may be  
revisited if the simulation objectives expand.

- **Hourly dispatch / intra-day scheduling** — Splitting the daily timestep into hourly or  
sub-hourly intervals to model solar peak vs evening demand curves. Requires hourly  
generation and demand profiles and significantly increases simulation complexity. The daily  
timestep is sufficient for seasonal energy balance and policy comparison.
- **Inverter efficiency curves** — Modeling inverter losses as a function of load factor rather  
than a flat system loss. Adds accuracy at partial load but requires detailed inverter specs  
and hourly resolution to be meaningful.
- **Battery degradation and cycle tracking** — Reducing usable battery capacity over time based
on cycle count, depth of discharge, and temperature. Requires sub-daily resolution for
meaningful cycle counting; the daily timestep cannot distinguish partial from full cycles.
Relevant for multi-year financial analysis but adds complexity with minimal impact on annual
energy balance.
- **Demand response / load shedding** — Deferring or curtailing non-critical loads (e.g.,  
warehouse climate control) during deficit periods. Requires load priority metadata and a  
controllable-load sub-model.
- **Time-of-use tariff optimization** — Shifting battery charge/discharge to exploit  
off-peak/peak price differentials. Requires hourly tariff schedules and intra-day dispatch,  
which conflicts with the daily timestep.
- **Electric vehicle charging** — Farm EVs or community transport as both load and mobile  
storage (V2G). Adds a vehicle fleet model and charging schedule layer.
- **Hydrogen electrolysis / fuel cell** — Using surplus renewable energy to produce hydrogen
